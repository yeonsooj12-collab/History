# 세계사 조건 렌즈 설계 메모

## 도구의 의미

세계사 조건 렌즈는 사회 압력, 외부 조건, 행위자, 영향을 받는 집단, 분류 기준, 통치 수단, 정당화 언어, 실제 변화, 저항·균열을 조합해 비교할 만한 세계사 쟁점을 찾는 도구다. 특정 역사 장면을 다른 장면과 같다고 단정하지 않고, 사건·제도·생활상·사조·예술·종교·관습의 닮은 점과 다른 점을 함께 본다.


## 앱 목적

사용자가 인간사회에 주어진 정치 조건, 사회 조건, 기술 조건, 내부 압력, 외부 압력 등을 직접 조합하면, 앱은 그 조합을 점수화하기보다 세계사와 비교할 수 있는 역사적 쟁점과 생활·문화 패턴을 찾는다. 목표는 자동으로 정답 사건을 뽑는 것이 아니라, 인간사가 반복해온 명분, 분류, 동원, 배제, 책임 이전, 생활 전략, 문화 표현, 저항과 균열의 패턴을 공부하도록 돕는 것이다.

## 핵심 설계 원칙

1. 여러 조건을 무작정 섞지 않는다.
2. 먼저 내부 압력과 외부 조건을 분리한다.
3. 행위자, 영향을 받는 집단, 분류 기준을 따로 본다.
4. 통치 수단, 정당화 언어, 실제 변화, 저항·균열을 함께 본다.
5. 데이터, 계산 엔진, 입력 정규화, 화면 제어를 분리한다.
6. `engine.js`는 DOM에 의존하지 않는 순수 함수 중심으로 유지하되 사용자-facing 주 엔진이 아니라 내부 참고와 fallback으로 낮춘다.
7. 기존 선택지와 직접 입력은 계산 엔진 앞에서 같은 option 구조로 취급한다.
8. AI는 평가자가 아니라 역사 해석 편집자다.
9. 낯선 조합은 우선 보존하고, 가능한 매개 논리와 역사적 비교 지점을 찾는다.
10. 사용자 원문은 자동으로 수정하거나 교체하지 않는다.
11. AI 제안과 사용자 입력의 출처를 구분한다.

## 상단 안내 구조

입력 UI 위에는 짧은 사용법 안내를 둔다. 이 영역의 목적은 사용자가 계산기의 사고 순서를 이해하게 하는 것이며, 실제 입력값을 바꾸거나 추천을 수행하지 않는다. 공식은 항상 노출하고, 예시는 기본적으로 접어 입력 UI까지의 스크롤 길이를 줄인다.

구성은 다음 순서다.

1. 계산기의 기본 공식
2. 기본 접힘 상태의 예시 튜토리얼
3. 빠른 시작 템플릿
4. 입력 UI로 넘어가는 전환 문구

공식은 3단계 프로세스로 보여준다.

1. ① 조건: 사회 압력, 외부 조건
2. ② 구조: 주도 행위자, 영향을 받는 집단, 분류 기준
3. ③ 작동과 결과: 통치 수단, 정당화 언어, 실제 변화, 저항·균열
4. 결과: 비교 가능한 세계사 렌즈

각 단계는 큰 컨테이너로 묶고, 단계 사이에는 큰 아래쪽 화살표를 둔다. 최종 결과 카드는 다른 색으로 강조한다. 상단 공식은 앱의 구조를 설명하는 용도이므로 짧게 유지한다. 모바일에서는 ① 조건 → ② 구조 → ③ 작동과 결과 → 결과 순서로 가로 스크롤 없이 세로 배치한다.

## 예시 제공 원칙

현재 제공하는 예시는 “식량 부족과 배급의 역사 렌즈” 하나다. 예시는 정답, 권장 설정, 추천 결과가 아니라 사용법 설명이다. 사용자가 예시를 펼치거나 접어도 실제 `formState`는 변경되지 않는다.

빠른 시작 템플릿은 예시와 다르게 실제 `formState`를 채운다. 템플릿은 `위기와 배급`, `국가가 사람을 세는 순간`, `이동과 국경`, `교육과 시민 만들기`, `생활상과 문화 변화`처럼 넓은 학습 질문을 여는 출발점이며, 특정 사건을 정답으로 추천하지 않는다. 시대·지역 범위는 사용자가 직접 정하도록 비워둔다.

예시의 기본 노출 범위는 작은 라벨, 제목, 짧은 설명, 주의 문구, `예시 전체 보기` 버튼이다. 기본 화면에서는 3단계 카드와 상세 비교 설명을 노출하지 않는다.

예시를 펼치면 단순 데이터 반복이 아니라 단계별 선택 이유를 설명하는 튜토리얼을 보여준다. 상세 영역은 다음 정보를 포함한다.

- STEP 1 조건: 무엇을 선택했고 왜 그렇게 선택했는가
- STEP 2 구조: 무엇을 선택했고 왜 그렇게 선택했는가
- STEP 3 작동과 결과: 무엇을 선택했고 왜 그렇게 선택했는가
- 비교 가능한 세계사 렌즈
- 비슷하다고 볼 이유, 같다고 보면 안 되는 이유, 더 봐야 할 문맥

사람을 묘사할 때는 존중하되, 제도가 가하는 배제·강제·차별은 정확하게 말한다. 역사 사례는 소재처럼 소비하지 않고 맥락과 차이를 함께 다룬다.

## 변수 그룹

- 조건: `pressure`, `technology`
- 구조: `actor`, `target`, `metric`
- 작동과 결과: `mechanism`, `ideology`, `transformation`, `careNarrative`

선택지는 정치·제도 사건만이 아니라 생활상, 대중문화, 예술 표현, 종교 의례, 물질문화까지 비교할 수 있도록 확장한다.

호환성을 위해 내부 JSON과 일부 함수명에는 과거 키(`technology`, `transformation`, `careNarrative` 등)가 남아 있다. 사용자 화면에서는 각각 외부 조건, 실제 변화, 저항·균열로 표시한다.

## 입력 상태 모델

UI는 현재 9개 표시 카테고리를 사용한다. 저장·복원과 과거 계약 호환을 위해 숨겨진 과거 키가 직렬화 과정에 남을 수 있다.

```js
{
  pressure: {
    mode: "preset",
    optionId: "",
    customText: "",
    metadata: {}
  }
}
```

- `mode`: `"preset"` 또는 `"custom"`
- `optionId`: 기존 선택지 ID
- `customText`: 직접 입력 문장
- `metadata`: 이후 확장을 위한 보조 정보

모드 전환은 기존 선택지와 직접 입력 값을 삭제하지 않는다. 사용자가 잠시 다른 모드로 갔다 돌아와도 입력 맥락이 보존되어야 한다.

## UI 상태 모델

입력값이 아닌 화면 상태는 별도의 UI state에 둔다.

```js
{
  exampleExpanded: false,
  structureInfoExpanded: false
}
```

예시 상세 접기/펼치기는 이 UI state만 바꾸며, 입력 `formState`, 계산 결과, 선택지 값, 직접 입력 텍스트를 변경하지 않는다. 또한 예시 토글은 입력 카드 DOM을 다시 생성하지 않고 해당 예시 패널의 `hidden`과 버튼 속성만 갱신한다.

AI 해석 상태는 별도의 `aiState`로 둔다.

```js
{
  status: "idle",
  input: null,
  response: null,
  error: null,
  isMock: true,
  isStale: false
}
```

입력 변경 시 기존 AI 응답은 stale 상태가 되지만 사용자 입력은 유지한다. reset 시 `aiState`는 초기값으로 돌아가고 구조 정보는 접힌다.

## 현재 UI 범위

현재 화면은 안내, 입력, mock AI 해석 표시까지 담당한다.

- 조건 섹션은 항상 열린다.
- 구조와 작동과 결과 섹션은 선택 영역이며 접고 펼칠 수 있다.
- 탐색 범위, 구조, 작동과 결과는 보조 힌트다. 사용자는 조건 하나만 적거나 골라도 1차 요청을 만들 수 있어야 하며, 진행 표시는 `0/n`처럼 필수 체크리스트로 보이지 않게 한다.
- 각 입력 카드는 질문, 설명, 기존 선택지, 직접 입력, 선택 설명, 추론 태그, 상태 메시지를 가진다.
- `ChatGPT에 세계사 렌즈 요청 만들기` 버튼을 눌렀을 때만 `evaluateResolvedInputs(formState)`를 호출한다.
- 평가 결과는 `createEvaluationResult(formState)`에서 `buildInterpretation(...)`, `buildAiInput(...)`, `createMockAiResponse(...)`와 결합한다.
- 입력 변경 후 기존 결과는 유지하되, 오래된 결과라는 안내를 표시한다.
- `초기화`는 입력 상태와 결과를 기본값으로 되돌리지만, 예시 안내는 입력 데이터와 분리된 설명 자료로 남는다.

## 결과 표시 원칙

결과는 점수표가 아니라 편집 노트다. 점수 계산은 `engine.js`와 `input.js`가 담당하지만, 기본 화면에는 compatibility를 표시하지 않는다. 규칙 기반 평가와 해석은 AI 입력의 내부 참고와 AI 실패 시 fallback으로 사용한다. 결과 화면은 progressive disclosure를 따른다.

- 기본 결과는 AI 상태 badge, 짧은 핵심 인사이트, 세계사 렌즈 본문을 우선 노출한다. 닮은 점과 다른 점, 반복되는 긴장, 질문, 대안 렌즈, 입력 구분은 접힌 상세로 이동한다.
- mock 응답과 추후 실제 AI 응답, fallback 응답을 badge로 구분한다.
- 구조 정보는 기본적으로 접힌 상태로 둔다.
- 구조 정보 안에 입력 누락, 규칙 기반 판단 신뢰도, 판단 근거 범위, 태그 정보, 규칙 기반 질문, compatibility, scatterRisk, 입력 요약을 둔다.
- compatibility는 아이디어 품질 점수가 아니라 태그 기반 참고 정보로만 설명한다.
- 배열형 출력은 화면에서 개수를 제한한다. 질문은 최대 4개, 대안 해석은 최대 2개다.
- 모든 사용자 표시값은 안전한 문자열 변환 helper를 거친다.
- 내부 객체를 직접 렌더링하지 않는다.
- AI 응답에는 creativityScore, qualityScore, compatibilityScore, realismScore, grade, ranking, correct, incorrect 같은 점수/등급 필드를 넣지 않는다.
- 사용자가 입력하지 않은 내용은 확정 사실이 아니라 가능한 해석, 연결 가설, 추가로 정할 조건으로 표시한다.

신뢰도가 낮으면 높은 점수처럼 보이는 수치보다 “판단할 정보가 충분하지 않다”는 메시지를 우선한다.

해석 엔진의 진단 상태는 `strong`, `workable`, `needs-detail`, `uncertain`, `missing`, `caution` 중 하나다. 표현은 사용자의 아이디어를 단정적으로 탈락시키는 말이 아니라, 어떤 연결을 더 설명하면 좋은지 안내하는 방식으로 유지한다.

## AI 데이터 계약

`js/ai.js`는 실제 네트워크 요청 없이 다음 계약을 정의한다.

- `buildAiInput(...)`: 사용자 선택, source, ruleContext, userIntent를 JSON 직렬화 가능한 데이터로 만든다.
- `createMockAiResponse(...)`: API 연결 전 화면 검증용 응답을 만든다.
- `createFallbackAiResponse(...)`: AI 실패 또는 응답 검증 실패 시 기존 규칙 기반 interpretation으로 응답을 만든다.
- `normalizeAiResponse(...)`: 화면 표시 전에 응답을 안전한 문자열과 제한된 배열로 정리한다.
- `validateAiResponse(...)`: status, headline, centralContradiction, questions, alternativeReadings, generatedSuggestions 구조를 검증한다.

현재 단계의 mock 응답은 실제 AI 품질을 흉내 내는 생성기가 아니라 UI와 데이터 흐름 검증용이다. 실제 API 연결 전까지는 `API 연결 전 미리보기` badge를 표시한다.

## Prompt Compiler Architecture

기본 사용자 흐름은 자동 API 호출이 아니라 수동 ChatGPT 연결이다. 앱은 `js/prompt.js`에서 로컬 편집 브리프를 만들고, 사용자가 복사할 수 있는 일반 ChatGPT용 전체 요청문을 생성한다.

- 로컬 브리프는 최종 해석이 아니다.
- 로컬 규칙은 창의적 연결을 판정하지 않는다.
- `engine.js`의 compatibility 숫자는 기본 화면과 전체 프롬프트에 노출하지 않는다.
- 낯선 연결은 오류가 아니라 AI가 해석해야 할 unresolved connection으로 보존한다.
- ChatGPT가 반환한 JSON은 `parsePastedAiResponse(...)`, `normalizeAiResponse(...)`, `validateAiResponse(...)`를 거쳐 기존 AI 결과 UI에 표시한다.
- 적용된 수동 응답의 source는 `manual-chatgpt`다.
- 선택적 API 개발 모드는 `?apiAi=1`, 개발용 mock은 `?mockAi=1`에서만 사용한다.

로컬 브리프와 AI 결과는 시각적으로 구분한다. 로컬 영역은 `AI에게 보낼 편집 브리프`, AI 결과 영역은 ChatGPT가 찾아낸 역사적 쟁점 또는 세계사 렌즈 출처로 표시한다.

### Axis Finder 단계

1차 ChatGPT 응답은 `interpretationMode: "axis-finder"`를 사용한다. 이 응답은 완성 설정이 아니라, 입력 요소에서 세계사와 비교할 수 있는 역사적 쟁점 2~3개를 분리한 결과다.

- 각 쟁점은 중심 모순 하나만 가진다.
- 각 쟁점은 `usedElements`, `deferredElements`, `unresolvedElements`를 구분한다.
- 모든 요소를 하나의 해석에 강제 통합하지 않는다.
- 보류 요소는 실패가 아니라 다른 쟁점이나 다음 역사 렌즈 단계의 재료다.
- 사용자가 쟁점을 선택한 뒤에만 2차 학습 프롬프트를 만든다.

수동 워크플로에서 앱에 다시 붙여 넣는 응답은 1차 `axis-finder` JSON뿐이다. 2차 답변은 ChatGPT 창에서 바로 읽는 해설로 받는다.

## 실제 API 연결 단계

브라우저 코드는 OpenAI API 키를 보관하지 않는다. 실제 해석 요청은 로컬 Node 서버의 `/api/interpret`로 보내고, 서버가 `.env`의 `OPENAI_API_KEY`를 읽어 OpenAI Responses API를 호출한다.

- 기본 모드: `세계사 렌즈로 읽기`
- 기본 결과 화면: `역사적 유사성`, `세계사 렌즈로 읽기`
- 접힌 상세: `무엇이 닮았고 무엇이 다른가`, `반복되는 긴장`, 질문, 다른 역사 렌즈, 입력 구분
- 접힌 상세: 질문, 다른 역사 렌즈, 입력과 AI 제안 구분
- 실패 처리: API 오류, 키 누락, 응답 검증 실패 시 `createFallbackAiResponse(...)`
- 개발용 mock: `?mockAi=1`일 때만 `createMockAiResponse(...)` 사용

OpenAI 요청은 `js/ai.js`의 `AI_RESPONSE_JSON_SCHEMA`와 `buildOpenAiStructuredRequest(...)`를 사용해 구조화된 JSON 응답을 요구한다. 규칙 기반 `engine.js`와 `interpretation.js`는 사용자에게 점수 판정을 내리는 주 엔진이 아니라, AI에 전달할 입력 구조와 fallback 진단을 제공하는 내부 보조 계층이다.

## 2차 학습 프롬프트

1차 `axis-finder` JSON을 적용하면 역사적 쟁점 카드가 나타난다. 사용자가 한 쟁점의 `이 쟁점을 ChatGPT에서 더 읽기`를 누르면 같은 결과 화면 안에 선택한 쟁점 요약과 2차 학습 프롬프트 복사 버튼이 열린다.

이 패널에는 2차 응답 붙여넣기 textarea가 없다. ChatGPT의 긴 해설, 참고 키워드, 후속 질문은 ChatGPT 창에서 이어서 읽는다.

2차 패널에서 할 수 있는 일:

- `2차 학습 프롬프트 다시 복사`: 현재 선택된 쟁점의 2차 요청문을 다시 복사한다.
- `쟁점 선택 취소`: 선택 상태를 초기화하고 1차 쟁점 카드 화면으로 돌아간다.

## 10A: 사회·정치·제도 분석 강화

이번 단계의 목적은 ChatGPT 프롬프트의 사고 순서를 강화하고, 그 결과를 세계사 공부로 연결하는 것이다. 앱은 사용자 입력을 정리하고, 요청문을 만들고, JSON을 붙여 넣고, 결과를 표시하는 도구로 남는다. 사회·정치·제도 조건, 그리고 역사적 유사 패턴을 해석하는 일은 ChatGPT가 맡는다.

프롬프트의 기준은 근거가 있는 사실, 검증 가능한 비교, 닮은 점과 다른 점의 구분이다. 출력 글자 수를 늘리는 것이 아니라, 결론을 만들기 전에 시대·지역·행위자·제도·결과를 분리해 검토하고, 불확실한 사건명이나 연도는 만들지 않는다. 확인하지 않은 참고문헌 번호, URL, 책 제목, 논문 제목, 기사 제목, 저자명, 쪽수, 직접 인용문도 만들지 않는다. 필요한 경우에는 가짜 각주 대신 사용자가 직접 검증할 검색 키워드를 남긴다. 1차 프롬프트는 JSON Schema의 모든 required 필드를 빠짐없이 채우라고 명시하며, 필드 누락 시 앱에 적용되지 않는다고 경고한다.

1차 `axis-finder`는 현실의 출발점, 사회문제와 대상, 행위자의 이해관계, 실제 권한 경로, 외부 조건의 실제 역할, 제도 변환 경로, 비용 이전, 비대칭, 중심 모순, 역사 렌즈 번역 순서로 생각하도록 지시한다.

2차 학습 프롬프트는 선택된 쟁점만 세계사 렌즈로 읽게 한다. JSON을 요구하지 않고, 핵심 요지, 비교할 세계사 장면, 앞뒤 맥락, 닮은 점과 결정적인 차이, 확인할 키워드, 더 공부할 질문을 사람이 읽는 해설로 요구한다.

1차 JSON 구조는 `axis-finder`의 `interpretationMode`와 기존 핵심 필드를 유지하되, 역사 공부에 필요한 구조 필드를 추가한다. 추가 필드는 `historicalCases`, `similarities`, `differences`, `humanBehaviorPattern`, `futureInsight`, `verificationKeywords`다. 수동 복사·붙여넣기 흐름은 바꾸지 않는다.

## 10C: 출력 압축과 계약 미세조정

10C는 새 기능이나 UI 확장이 아니라 배포 전 프롬프트 출력 품질을 압축하는 단계다. 기존 JSON 필드는 유지하고 필드별 작성 분량만 제한한다.

1차 `axis-finder` 분량 제한:

- `overview`: 최대 4문장
- `title`: 한 문장
- `coreInsight`: 최대 3문장
- `centralContradiction`: 한 문장
- `bridge`: 최대 5문장
- `assumptions`: 최대 2개
- `whyThisAxisMatters`: 최대 5문장
- `specificQuestion`: 한 문장
- `historicalCases`: 1~4개. 각 항목은 `title`, `period`, `region`, `context`, `insight`, `verificationKeywords`를 포함
- `similarities`: 최대 4개
- `differences`: 최대 4개
- `humanBehaviorPattern`: 최대 4문장
- `futureInsight`: 최대 4문장
- `verificationKeywords`: 3~8개
- `unresolvedElements`: 최대 5개
- `editorNote`: 최대 3문장

2차 학습 해설 형식:

- 핵심 요지
- 비교할 세계사 장면 2~4개
- 앞뒤 맥락
- 닮은 점과 결정적인 차이
- 확인할 키워드 5~8개
- 더 공부할 질문 3~5개

앱에 붙여 넣는 신규 응답의 `version`은 `"1.0"`으로 고정한다. 앱에 붙여 넣는 `interpretationMode`는 `axis-finder`만 사용한다.

시대·지역 범위 입력은 `0단계 · 탐색 범위`로 분리한다. 규칙 기반 점수 계산의 필수 변수로 쓰지 않고, 사용자가 세계사 비교 범위를 좁히고 싶을 때 ChatGPT 프롬프트에 전달하는 보조 힌트로만 사용한다. 비워두면 전 시대·전 지역에서 비교 후보를 찾는다는 뜻이다.

사이트 안에서 실제 검색/API 기반 출처 검증 루프는 제공하지 않는다. 현재 제품은 AI가 가짜 각주를 만들지 않게 막고, 각 사례마다 검증 키워드를 남기게 한 뒤, 사용자가 별도 검색으로 직접 확인하는 방식이다. 따라서 앱은 확정된 참고문헌을 보증하는 도구가 아니라, 검증 가능한 역사 학습 질문과 검색 단서를 만드는 도구다. 결과 화면은 쟁점별 `확인 키워드 복사`를 제공해 수동 검증을 쉽게 만든다. 쟁점 카드 상단에는 비교 가설 안내를 표시해 사건명·연도·원인관계를 별도 자료로 확인해야 함을 알린다.

붙여 넣은 ChatGPT 응답이 최신 `axis-finder` 구조를 따르지 않으면 앱은 누락 필드 일부를 보여주고 `다시 요청 문구 복사`를 제공한다. 이 복구 문구는 표준 JSON 큰따옴표, JSON 앞뒤 설명 금지, `historicalCases`, `similarities`, `differences`, `humanBehaviorPattern`, `futureInsight`, `verificationKeywords` 필수 포함을 다시 요구한다.

생성 과정 메타 문장은 모든 문자열 필드에서 금지한다. 금지 예시는 “제공된 편집 브리프에 근거한다”, “선택된 방향의 범위를 따른다”, “업로드된 자료에 맞추었다”, “원본 브리프와 정렬했다”, “source alignment”, “사용자 입력에 충실하게 작성했다”, “요청된 JSON 구조를 따랐다”, “이 응답은 앞 단계 결과를 기반으로 한다”다.

`generatedSuggestions`는 최대 3개로 제한하고 서로 다른 type만 사용한다. 비교할 역사 사건이나 제도는 `possibleHistoricalEvent`, `possibleInstitution`, `possibleComparisonKeyword`, `possibleDifference` 같은 type으로 적는다. 행정 문구는 `possibleAdministrativeLanguage` type으로 최대 1개만 작성한다. 금지 type은 `sourceAlignment`, `deferredElement`, `addedAssumption`, `administrativeLanguage`다.

입력 사회문제를 인접 개념으로 조용히 대체하지 않는다. 인접 문제가 꼭 필요하면 `assumptions`, `possibleBridge.assumptions`, 또는 `cautions`에 명시한다. 사회문제를 배경에서 한 번 언급한 것만으로 사용했다고 판정하지 않는다.

## 10D: 인과관계 우선순위와 억지 브리지 방지

10D 이후의 원칙은 강한 결론이나 유명한 역사 비유를 유지하기 위해 현실적으로 약한 중간 제도를 새로 발명하는 문제를 막는 것이다. 원칙은 “낯설음은 결론에서 만들고, 그 결론까지 가는 인과경로와 역사 비교 기준은 현실적이고 단순하게 만든다”이다.

절대 우선순위는 중심 인과관계의 명확성, 현실적인 권한·비용·기술 경로, 사회문제와 대상의 충실한 사용, 나머지 입력 요소의 사용률 순서다. 하위 기준을 만족하기 위해 상위 기준을 훼손하지 않는다.

약한 브리지의 징후는 입력에 없는 대규모 정책 전환, 서로 다른 재정·보험·법률 체계의 무리한 통합, 인접 사회문제로의 조용한 이동, 새로운 기관·기금·면허·급여체계 발명, 설명할수록 늘어나는 가정이다.

중대한 가정은 `assumptions`로 통과시키지 않는다. 예를 들어 연금 현금급여를 장기요양 서비스로 대체하거나, 건강보험과 연금 재정을 하나로 통합하거나, 기업이 사실상 국가의 허가권을 갖는 것은 경미한 보조 가정이 아니다.

각 axis는 내부적으로 한 문장 인과사슬이 성립해야 한다. “현실의 압력 때문에 행위자 A가 자신의 이해관계에 따라 수단 B를 사용하고, 그 결과 대상 C에게 변화 D가 발생하며, 그 논리를 과잉 적용하면 모순 E가 생긴다.” 이 문장을 새 제도 이름 없이 쓸 수 없으면 방향을 다시 작성하거나 보류한다.

사회문제 사용 강제는 완화한다. 모든 axis는 대상을 중심에 두지만, 사회문제는 실제 중심 인과관계에 참여할 때만 `usedElements`에 넣는다. 2~3개 방향 중 최소 1개는 입력 사회문제를 직접 사용하는 현실적 방향을 우선 탐색하되, 현실적 경로가 없으면 사회문제를 억지로 사용하는 axis를 만들지 않는다.

2차 `editor`는 선택한 axis의 약한 연결을 장문 분석이나 거대한 역사 비유로 숨기지 않는다. 약한 연결이 발견되면 해당 요소를 중심 비교에서 제외하고, 필요하면 `cautions`에 내용상의 한계 설명을 짧게 남긴다. 생성 과정에 대한 메타 문장은 계속 금지한다.
